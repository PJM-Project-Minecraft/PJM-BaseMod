# Веб-панель PJM BaseMod — дизайн

Дата: 2026-07-04. Статус: одобрен пользователем в брейнсторминге.

## Цель

Встроенная в мод веб-панель для админов сервера: живые графики TPS/MSPT/heap/онлайна
(в духе Spark), мониторинг entity с координатами и замером нагрузки, список игроков,
инструменты модерирования (варны/баны/муты через существующий `ModerationService`,
кик, телепорт, удаление entity). Доступ через интернет, вход через игру.

## Ключевые решения (зафиксированы с пользователем)

| Вопрос | Решение |
|---|---|
| Доступ | Публичный интернет → полноценная авторизация |
| Вход | `/pjm web login` в игре → одноразовый код → сессия |
| Уровни доступа | Один уровень: полный доступ у всех вошедших |
| Модерация из панели | Варны/баны/муты, кик, телепорт, удаление entity. **Без** консоли произвольных команд |
| Нагрузка entity | Реальный замер тика через `EntityTickEvent.Pre/Post`, тумблер |
| HTTP-сервер | **Javalin 6** через NeoForge jarJar (Jetty, WebSocket) |
| Фронтенд | **React 18 + Vite 5** (Node 18 на машине), собранный `dist` коммитится в ресурсы |
| Координаты | Таблицы + мини-карта точек X/Z (canvas, зум/пан), без рендера рельефа |
| TLS | Не в моде — reverse proxy (nginx/caddy), задокументировать |

## Архитектура

Новая подсистема `src/main/java/ru/liko/pjmbasemod/common/web/` — целиком server-side.
Клиентский source set не затрагивается, MC-пакеты не добавляются, `PjmNetworking.VERSION`
не бампается.

```
common/web/
├── WebPanelService.java     — lifecycle: старт Javalin на ServerStartedEvent,
│                              стоп на ServerStoppingEvent; ошибки не роняют сервер
├── WebAuthService.java      — одноразовые коды входа, обмен на сессию, троттлинг
├── WebSessions.java         — сессионные токены в памяти, TTL, привязка к нику/UUID
├── WebActions.java          — мост к игре: kick/teleport/ban/warn/mute/remove entity
│                              (всё через server.execute + CompletableFuture)
├── metrics/
│   ├── MetricsCollector.java — сэмплы 1/сек в ring buffer
│   ├── MetricsSample.java    — record одного сэмпла
│   └── EntityProfiler.java   — замер тика entity, скользящее окно 30 c
└── api/
    ├── WebApiRoutes.java     — REST-эндпоинты
    ├── WebSocketHub.java     — пуш живых кадров подключённым браузерам
    └── WebJson.java          — DTO/сериализация (Gson, уже в classpath)
```

### Модель потоков (критично)

Jetty-потоки Javalin **никогда не читают и не пишут игровое состояние напрямую**:

- **Чтение**: `MetricsCollector` и снапшоты entity/игроков собираются на server tick
  в иммутабельные структуры (`volatile`-ссылка на готовый снапшот / concurrent ring
  buffer). HTTP-потоки читают только их. Тяжёлый список всех entity сериализуется
  заранее на тике (раз в 1–2 сек), а не по запросу.
- **Запись (действия)**: HTTP-хендлер → `server.execute(...)` → `CompletableFuture`,
  ожидание с таймаутом 5 с. Таймаут → HTTP 504.

### Конфиг — новая секция `web` в `Config.java`

| Ключ | Default | Смысл |
|---|---|---|
| `web.enabled` | `false` | панель выключена по умолчанию |
| `web.port` | `8776` | порт HTTP |
| `web.bindAddress` | `0.0.0.0` | адрес привязки |
| `web.sessionTtlMinutes` | `720` | TTL сессии (12 ч) |
| `web.historyMinutes` | `120` | глубина истории графиков |
| `web.profilerEnabled` | `true` | разрешён ли профайлер entity вообще |
| `web.publicUrl` | `""` | базовый URL для кликабельной ссылки логина; пусто → печатаем только код |

### Сборка (build.gradle)

- `jarJar(implementation("io.javalin:javalin:6.6.0"))` (последняя 6.x на момент
  имплементации — сверить на Maven Central) — тянет Jetty 11 + kotlin-stdlib,
  ~4–5 МБ к JAR. Проверить, что dedicated-server JAR получает вложенные jar-ы.
- Gson уже доступен из Minecraft — новых JSON-зависимостей нет.

## Авторизация

1. `/pjm web login` (подкоманда в `PjmCommands`, permission level 3+).
2. Сервер генерирует одноразовый код (8 симв., TTL 5 мин) и печатает кликабельную
   ссылку `http://<host>:<port>/login?code=XXXX` (host — из конфига `web.publicUrl`,
   опциональный ключ; если пуст — печатаем только код и подсказку).
3. `POST /api/auth/exchange {code}` → сессионный токен 256 бит → HttpOnly-cookie,
   `SameSite=Strict`.
4. Сессии в памяти; `/pjm web logout` (снять свои) и рестарт сервера сбрасывают все.
5. Каждое действие пишется в server log: `[WebPanel] <ник админа> banned <игрок> ...`.
6. Троттлинг `/api/auth/*`: 5 попыток/мин с одного IP.
7. CSRF: SameSite=Strict + проверка заголовка `X-Requested-With` на POST.

## Метрики

`MetricsCollector` — аккумулирует на каждом тике, фиксирует сэмпл раз в секунду
в кольцевой буфер (`historyMinutes` × 60 сэмплов):

- TPS и MSPT: общий + по дименшенам (avg/min/max за секунду);
- Heap: used/max;
- Онлайн игроков;
- Entity: количество по дименшенам и категориям (мобы / item / техника / прочее);
- Загруженные чанки по дименшенам.

`EntityProfiler` — выключен по умолчанию, тумблер из панели (если `web.profilerEnabled`).
Подписан на `EntityTickEvent.Pre/Post`, первая строка хендлера — проверка `enabled`.
`System.nanoTime()` до/после тика. Скользящее окно 30 с:

- топ-100 тяжёлых entity: тип, имя, UUID, координаты, дименшен, мкс/тик, % от MSPT;
- агрегация по типам (`minecraft:zombie — 12.3%`);
- агрегация по чанкам («горячие» чанки для мини-карты).

Оверхед: ~2×50 нс на entity; при 5000 entity ≈ 0.25 мс/тик — приемлемо для
временного включения.

## API

Все `/api/*` (кроме `auth`) требуют валидную сессию, иначе 401.

| Метод | Путь | Действие |
|---|---|---|
| POST | `/api/auth/exchange` | код → сессия |
| POST | `/api/auth/logout` | завершить сессию |
| GET | `/api/overview` | текущее состояние + история для графиков |
| GET | `/api/players` | ник, UUID, координаты, дименшен, пинг, фракция, роль, активные наказания |
| GET | `/api/entities?dim=&type=&page=` | снапшот entity с координатами, пагинация |
| GET | `/api/profiler` | данные профайлера |
| POST | `/api/profiler/toggle` | вкл/выкл профайлер |
| GET | `/api/moderation/history?player=` | история наказаний из `ModerationSavedData` |
| POST | `/api/actions/kick` | кик с причиной |
| POST | `/api/actions/warn` `/ban` `/mute` `/pardon` | вызовы `ModerationService`, инициатор — ник из сессии |
| POST | `/api/actions/teleport` | игрока к координатам или к игроку |
| POST | `/api/actions/entities/remove` | по списку UUID или массово (тип + дименшен + опц. радиус) |

WebSocket `/ws/live` (сессия проверяется при upgrade): сервер пушит раз в секунду
компактный кадр (TPS/MSPT/heap/онлайн + дельты игроков), раз в 2 с — entity-статистику.
Браузер шлёт только ping. Все действия — через REST.

Массовое удаление entity и бан требуют подтверждения на фронте (двухшаговая кнопка).
Игроки (`Player`) массовым удалением не затрагиваются никогда.

## Фронтенд

- Исходники: `web/` в корне репозитория — React 18 + Vite 5 + TypeScript.
  Node 18.19 на машине разработчика (Vite 5 совместим; Vite 7 нельзя — нужен Node 20+).
- Графики: **uPlot** (лёгкий, тянет 7200 точек без лагов). Анимации UI: **framer-motion**.
- Сборка: `npm run build` → `vite build` с `outDir=../src/main/resources/web`,
  **dist коммитится** — `./gradlew build` работает без Node.
- Javalin отдаёт статику из classpath `/web`, SPA-fallback на `index.html`.

Тёмная «тактическая» тема под стиль мода: приглушённый фон, акценты цветов фракций,
моноширинный шрифт для координат/чисел, framer-motion-переходы вкладок, анимированные
счётчики, пульсирующий индикатор соединения WS.

Вкладки:
1. **Дашборд** — живые графики TPS/MSPT/heap/онлайн со скроллом по истории, карточки-сводки;
2. **Игроки** — таблица (поиск/сортировка), карточка игрока: действия + история наказаний;
3. **Entity** — сводка по типам, топ тяжёлых (профайлер), таблица с фильтрами, удаление с подтверждением;
4. **Карта** — canvas X/Z: игроки (цвет = команда scoreboard), entity (цвет = категория),
   зум/пан, тултипы, подсветка горячих чанков из профайлера.

Язык интерфейса панели — русский (единственный; i18n панели вне скоупа).

## Обработка ошибок

- Экшены отвечают JSON `{ok, error}`; таймаут server thread → 504; игрок оффлайн /
  entity исчез → 404 с сообщением; нет сессии → 401 (фронт показывает экран
  «войдите через /pjm web login»).
- Ошибка старта Javalin (порт занят и т.п.) → лог `ERROR`, панель отключена,
  сервер продолжает работать.
- Разрыв WS на фронте → авто-reconnect с экспоненциальной задержкой, индикатор
  состояния соединения.

## Верификация

- `./gradlew compileJava` — сборка бэкенда; `npm run build` в `web/` — сборка фронта.
- Внутриигровую и браузерную проверку выполняет пользователь из каталога без `!`
  (dedicated server + браузер → `localhost:8776`).
- Ручной чек-лист: login-flow, все 4 вкладки, каждое действие модерации, поведение
  при рестарте сервера (сброс сессий), троттлинг кода.

## Вне скоупа

- Консоль произвольных команд (RCON) — отклонено сознательно.
- Рендер рельефа мира (BlueMap уже интегрирован).
- Sampling-профайлер стеков как в настоящем Spark.
- TLS внутри мода, i18n панели, гранулярные права.
