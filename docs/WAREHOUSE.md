# Система склада (Warehouse)

Склад — экономика снабжения команды. Каждый **именованный склад** копит очки в двух
**пулах**. Игроки получают снаряжение у **NPC-кладовщика**, тратя очки пула, и могут
сдавать снаряжение обратно за частичный возврат. Очки пополняются **ящиками поставок**,
которые сбрасывают в **зону приёма** склада или вручают NPC, а пул `supply` — ещё и
**пассивным доходом с удерживаемых точек захвата**.

- Серверная логика: `common/warehouse/WarehouseManager`.
- NPC: `common/entity/QuartermasterEntity` (рендер `QuartermasterRenderer`).
- Клиентский экран: `client/gui/screen/WarehouseScreen`.
- Команды: `/pjm warehouse …` (`common/command/WarehouseCommands`).

## Пулы очков

Очки склада разделены на **2 пула** (`WarehousePoolCategory`). Каждый ящик пополняет
один пул, каждый выдаваемый предмет списывает очки из своего пула:

| Пул | id | Чем пополняется |
|-----|-----|-----------------|
| Припасы | `supply` | Ящики, сдача снаряжения, пассивный доход с точек захвата |
| Сырьё | `raw` | Только ящики и сдача |

**Устаревшие пулы.** Раньше пулов было пять; `weapon`, `equipment` и `special` схлопнуты
в `supply`. Править конфиги не нужно:

- в `items.json` и `crates/*.json` старые значения `pool` читаются как алиасы `supply`
  (`WarehousePoolCategory.byId`) — но в автодополнении команд их больше нет;
- в сохранённых мирах очки старых пулов **суммируются** в `supply` при первой загрузке
  (`WarehouseSavedData.load`), ничего не теряется;
- `displayCategory` — **не** пул, а вкладка GUI: `weapon`/`equipment`/`special` там
  остаются рабочими значениями.

### Пассивный доход с точек захвата

Раз в минуту каждая команда получает в пул `supply` своего склада
`incomePerPointPerMinute` очков **за каждую удерживаемую точку захвата**. Начисление идёт
только при активном захвате (`capturePoints.enabled`, в т.ч. по расписанию). Секция
`capturePoints` в `config.json`:

```jsonc
"capturePoints": {
  "incomeEnabled": true,
  "incomePerPointPerMinute": 10,
  // teamId (секция teams) → warehouseId (/pjm warehouse create).
  // Команды без записи здесь дохода не получают.
  "warehouseByTeam": {
    "team1": "wh_red",
    "team2": "wh_blue"
  }
}
```

Пример: команда `team1` удерживает 3 точки → её склад `wh_red` получает 30 очков `supply`
в минуту. Склад создаётся автоматически, если его ещё нет (как при приёмке ящиков), —
опечатка в `warehouseByTeam` проявится лишним складом в `/pjm warehouse list`.

## Каталог предметов `items.json`

Предметы, доступные у кладовщика, задаются в серверном файле:

```text
config/pjmbasemod/warehouse/items.json
```

Если файла нет, мод создаёт пример при старте сервера. Поддерживается и **legacy-формат** —
отдельные файлы `config/pjmbasemod/warehouse/items/<id>.json`. Если один и тот же `id` есть
и в legacy-папке, и в `items.json`, запись из `items.json` заменяет старую.

### Формат

```json
{
  "schemaVersion": 1,
  "items": [
    {
      "id": "ak74m",
      "displayName": "АК-74М",
      "itemId": "superbwarfare:ak_47",
      "pool": "weapon",
      "displayCategory": "weapon",
      "pointCost": 2,
      "quantity": 1,
      "maxPerWithdraw": 1,
      "refundValue": 1,
      "allowedRoles": ["assault", "sso"],
      "allowedTeams": ["team1"],
      "minRank": "private"
    },
    {
      "id": "medkit",
      "displayName": "Аптечка",
      "itemId": "minecraft:golden_apple",
      "pool": "supply",
      "displayCategory": "medicine",
      "pointCost": 1,
      "quantity": 4,
      "refundValue": 1
    }
  ]
}
```

### Поля

| Поле | Тип | По умолчанию | Что значит |
|------|-----|--------------|------------|
| `id` | string | — | Уникальный id записи. Латиница, цифры, `_`, `-`; остальное нормализуется. |
| `displayName` | string | id предмета | Имя в GUI. Если пусто — клиент подтянет локализованное имя предмета. |
| `itemId` | string | — | Registry id предмета (`minecraft:iron_ingot`, `superbwarfare:ak_47`) или TACZ gunpack id (`tacz:ak47`, `tacz:762x39`). Необязателен, если задан `itemNbt` или блок `tacz`. |
| `pool` | string | `supply` | Пул очков: `supply` / `raw`. Устаревшие `weapon`/`equipment`/`special` читаются как `supply`. |
| `displayCategory` | string | = `pool` | Вкладка GUI: `weapon`, `ammo`, `food`, `medicine`, `equipment`, `raw`, `vehicle`, `special`. |
| `pointCost` | int | `1` (минимум 1) | Стоимость **одной покупки** в очках пула. |
| `quantity` | int | `1` | Сколько штук выдаётся за одну покупку (за `pointCost` очков). |
| `maxPerWithdraw` | int | `16` (если `0`) | Верхняя граница пачки для GUI. |
| `refundValue` | int / `null` | = `pointCost` | Возврат очков за сдачу одной пачки (`quantity` штук). `0` — предмет нельзя сдавать; поле отсутствует — равно `pointCost`. |
| `allowedRoles` | string[] | все | Роли, которым доступна выдача (см. [роли](#роли)). Пусто/нет — доступно всем. |
| `allowedTeams` | string[] | все | Команды (scoreboard-id), которым доступна выдача. Чужим командам предмет не виден в каталоге. |
| `minRank` | string | без ограничения | Минимальный ранг (id), начиная с которого предмет доступен. |
| `permission` | string | все | Ключ донат-пермишена. Выдача требует ноду `pjmbasemod.warehouse.perm.<permission>` (выдаётся донат-плагином/LuckPerms). Один ключ у нескольких предметов = общая нода. Без бэкенда — доступно только OP. Предмет виден всем с замком «Доступно за донат». |
| `components` | string | — | Компоненты данных (NBT) в синтаксисе `/give`, напр. `"[minecraft:custom_name='АК',minecraft:max_stack_size=1]"`. Применяются поверх стека. |
| `itemNbt` | string | — | Полный SNBT предмета (`{components:{…},count:1,id:"ns:item"}`). Если задан — предмет выдаётся точь-в-точь; `components` игнорируется, `itemId` нужен лишь для иконки/совпадения. Обычно заполняется командой `additem`. |
| `tacz` | object | — | Декларативное описание ствола TACZ (см. [TACZ](#tacz)). Предпочтительнее ручного NBT. |
| `taczId` | string | — | Реальный id «простого» TACZ-предмета — патрона/обвеса (`tacz:762x39`, `tacz:red_dot_sight`). Нужен, т.к. у них один базовый Item на все варианты (см. [TACZ](#tacz)). `itemId` при этом не обязателен. |
| `roleLocked` | boolean | `false` | Запрещать применение/удержание предмета игроку не той роли из `allowedRoles`. |
| `lockMode` | string | `auto` | Режим роль-лока: `auto` (TACZ-оружие → удержание, иначе использование), `use`, `hold`. |

> **Одна покупка = ровно `quantity` штук за `pointCost` очков.** Массовой выдачи нет.
> Сдача принимается только целыми пачками по `quantity` штук — это симметрично выдаче и
> закрывает дюп очков.

### Нормализация при загрузке

- `pointCost ≤ 0` → `1`; `maxPerWithdraw ≤ 0` → `16`; `quantity ≤ 0` → `1`.
- Неизвестный `lockMode` → `auto` (с предупреждением в логе).
- Если все указанные `allowedRoles`/`allowedTeams` неизвестны — запись пропускается / ограничение снимается, причина пишется в лог.
- Неизвестный `minRank` → ограничение снимается (с предупреждением).
- Для декларативного `tacz` без `itemId` подставляется `tacz:modern_kinetic_gun` (для иконки и совпадения).
- Для `taczId` (патрон/обвес) `itemId` не обязателен — тип и базовый предмет определяются по индексам TACZ.

### Роли

`allowedRoles` принимает canonical id ролей (`CombatRole`):

```text
assault, machine_gunner, sniper, uav_operator, sso, pilot, ew_specialist, crew
```

### TACZ

Способы задать предметы TACZ:

**1. Ствол просто по id** (как обычный предмет):

```json
{
  "id": "ak47_tacz",
  "displayName": "AK-47",
  "itemId": "tacz:ak47",
  "pool": "weapon",
  "displayCategory": "weapon",
  "pointCost": 2,
  "quantity": 1
}
```

**2. Ствол — декларативный блок `tacz`** (читаемо, с патронами, режимом огня и обвесами):

```json
{
  "id": "ak74m_tacz",
  "displayName": "АК-74М",
  "pool": "weapon",
  "displayCategory": "weapon",
  "pointCost": 3,
  "quantity": 1,
  "tacz": {
    "gunId": "mk16:ak74m",
    "ammo": 30,
    "fireMode": "AUTO",
    "ammoInBarrel": true,
    "attachments": {
      "scope": "tacz:red_dot",
      "muzzle": "tacz:silencer"
    }
  }
}
```

Поля блока `tacz`:

| Поле | Что значит |
|------|------------|
| `gunId` | id ствола TACZ/ганпака, напр. `mk16:ak74m`, `tacz:ak47`. |
| `ammo` | Патронов в магазине. |
| `fireMode` | `AUTO` / `SEMI` / `BURST` (без учёта регистра). Пусто — режим ствола по умолчанию. |
| `ammoInBarrel` | Дослан ли патрон в ствол. `null` — авто (`ammo > 0`). |
| `attachments` | Обвесы по слотам: `scope`, `muzzle`, `stock`, `grip`, `laser`, `extended_mag` → id обвеса. |

**3. Патроны и обвесы — декларативный `taczId`.** Патроны (`IAmmo`) и обвесы (`IAttachment`)
в TACZ — это **один** базовый предмет (`tacz:ammo` / `tacz:attachment`) на все варианты; конкретный
патрон/обвес задаётся реальным id внутри NBT (как `gunId` у стволов). Поэтому задавать их «просто по
id предмета» нельзя — GUI покажет общий ключ `item.tacz.ammo` / `item.tacz.attachment` и missing-иконку.
Используй поле `taczId` с **реальным** id патрона/обвеса:

```json
{
  "id": "ammo_762x39",
  "pool": "supply",
  "displayCategory": "ammo",
  "pointCost": 1,
  "quantity": 30,
  "taczId": "tacz:762x39"
}
```
```json
{
  "id": "red_dot",
  "pool": "weapon",
  "displayCategory": "attachment",
  "pointCost": 15,
  "taczId": "tacz:red_dot_sight"
}
```

При `taczId` поле `itemId` не обязательно — склад сам определит тип (патрон/обвес) по индексам TACZ,
соберёт правильный предмет (патронам выставит размер стака), подтянет иконку и локализованное имя.

Если в старом конфиге стоит `superbwarfare:ak_47`, а самого SuperBWarfare-предмета нет,
склад попробует выдать TACZ-аналог (`tacz:ak47`).

**Самый надёжный способ** занести любой TACZ-предмет — взять его в руку и выполнить
[`/pjm warehouse additem`](#команды): команда сама распознаёт тип и захватывает декларативно —
ствол → блок `tacz` (с `gunId`, патронами, режимом огня, обвесами), патрон/обвес → `taczId`,
прочее → полный NBT.

## Ящики поставок `crates/`

Ящики настраиваются по одному файлу на тип:

```text
config/pjmbasemod/warehouse/crates/<id>.json
```

Формат (`CrateDefinition`):

```json
{
  "id": "weapon_crate",
  "pool": "weapon",
  "points": 3
}
```

| Поле | Что значит |
|------|------------|
| `id` | id ящика = path зарегистрированного предмета-ящика. |
| `pool` | Пул очков, который пополняет ящик. |
| `points` | Сколько очков даёт один ящик (минимум 1). |

Стандартные ящики (предметы `pjmbasemod:<id>`, стек до 16):

| Ящик | Пул | Очки |
|------|-----|------|
| `weapon_crate` | `supply` | 3 |
| `supply_crate` | `supply` | 5 |
| `equipment_crate` | `supply` | 4 |
| `raw_crate` | `raw` | 5 |
| `special_crate` | `supply` | 2 |

Пять предметов-ящиков сохранены (разный вес поставки), но все, кроме `raw_crate`,
теперь пополняют `supply`.

Сдать ящики на склад можно двумя способами:
- **ПКМ ящиком по NPC-кладовщику** — сдаётся весь стек в руке на склад, привязанный к NPC.
- **Бросить ящики в зону приёма** склада — они поглощаются автоматически (см. ниже).

## NPC-кладовщик

`QuartermasterEntity` — NPC без ИИ, неуязвимый, не толкается, не деспавнится.
**ПКМ по NPC** открывает GUI склада; **ПКМ ящиком** — сдаёт ящик.

Настройки хранятся на самой сущности (сохраняются в мире) и задаются командами
`/pjm warehouse npc …` для **ближайшего NPC** (радиус 12 блоков):

| Параметр | По умолчанию | Что значит |
|----------|--------------|------------|
| `warehouseId` | `""` | К какому именованному складу привязан (его очки и каталог). |
| `skinId` | `skin_emr` | Скин кладовщика. |
| `teamRestriction` | `""` (все) | Если задан — открыть склад может только эта команда. |
| `withdrawLimit` | `0` (из предмета) | Лимит выдачи за раз (`0…256`). |
| `cooldownTicks` | `0` | Задержка между выдачами, в тиках (`0…72000`; 20 тиков = 1 с). |
| `allowedCategories` | пусто (все) | Какие `displayCategory` выдаёт этот NPC. |

Доступные скины: `skin_emr`, `skin_mc`, `skin_atacsfg`, `skin_m05`,
`skin_emr+multicam`, `skin_emr+atacsfg`, `skin_emr+jaket`, `skin_mc+jacket`.

## Зона приёма поставок

У каждого склада может быть **зона приёма** — место, куда сбрасывают ящики для
автоматического зачисления очков (`WarehouseSettings`, сохраняется per-world):

| Параметр | По умолчанию | Что значит |
|----------|--------------|------------|
| Измерение + позиция | — | Точка, задаётся `/pjm warehouse reception set` (берётся позиция игрока). |
| Радиус | `4` | Радиус куба зоны в блоках (`1…128`), `/pjm warehouse reception radius`. |

Брошенные в зону предметы-ящики поглощаются на серверном тике, очки начисляются складу,
бросившему игроку приходит уведомление. Когда рядом есть игрок (в радиусе 48 блоков),
зона **подсвечивается частицами** `END_ROD` по периметру на уровне пола — видно, куда бросать.

## Личный бюджет (анти-«пылесос»)

`WarehousePersonalBudgetSavedData` — персональный лимит очков **на игрока**, чтобы один
человек не выкачал весь склад. Потолок и регенерация берутся из **ранга** игрока
(`WarehouseBudgetLimits.forPlayer`), а не из глобального конфига.

**Настройка по рангу** — поля в конфиге рангов (`config/pjmbasemod/ranks/…`, класс `RankDefinition`):

| Поле ранга | Значение |
|------------|----------|
| `warehouseBudgetMax` | Потолок квоты для этого ранга. **Отсутствует или `-1` → безлимит.** `>0` → лимит. |
| `warehouseBudgetRegenPerHour` | Восстановление, очков/час. **Отсутствует → равно потолку** (полное восстановление за час). |

Дефолтные ранги уже идут со scaling-бюджетами: рядовой `50` → майор `1000`.

Как работает:
- Бюджет **глобальный на игрока** — один на все склады и все пулы.
- При выдаче `pointCost` списывается **и со склада, и из личного бюджета**. Если личного
  бюджета не хватает — выдача блокируется, даже если очки на складе есть.
- Потолок = `warehouseBudgetMax` **текущего ранга** игрока; при повышении ранга потолок растёт.
- Новый игрок стартует с **полным** бюджетом (по потолку своего ранга).
- Бюджет **плавно регенерирует** до потолка. Регенерация ленивая — пересчитывается при
  обращении по серверному игровому времени overworld-а, дробный остаток не теряется.
- **Сдача снаряжения возвращает** очки и в личный бюджет (но не выше потолка).
- **OP** (уровень прав ≥ 2) и ранги с `warehouseBudgetMax = -1`/без поля → **безлимит**
  (в GUI показывается «∞»).

**Восполнить квоту вручную** — команда `/pjm warehouse budget <player> …` (см. [Команды](#команды)).

## Сдача снаряжения

`handleDeposit` — возврат снаряжения складу за очки в пул:

- Принимается, только если у предмета `refundValue > 0` (`depositable`).
- Сдаётся **целыми пачками** по `quantity` штук — симметрично выдаче (защита от дюпа).
- Для предметов **с прочностью** начисление масштабируется по остатку прочности: изношенный
  даёт меньше очков, «убитый в ноль» — `0` (и не списывается). Многоразовые (без прочности) — полную ставку.
- Возврат за пачку округляется **вниз** (без переплат): `floor(refundValue × Σпрочность / quantity)`.
- Сдача восстанавливает и личный бюджет (если у игрока действует лимит, т.е. не безлимит).

## Команды

Все под `/pjm warehouse …`, требуют **OP уровня 2**.

### Склады

| Команда | Действие |
|---------|----------|
| `create <id>` | Создать именованный склад. |
| `delete <id>` | Удалить склад и его настройки. |
| `list` | Список складов. |

### Очки

| Команда | Действие |
|---------|----------|
| `points add <id> <pool> <amount>` | Добавить очки в пул (`amount` 1…100000). |
| `points get <id>` | Показать очки всех пулов склада. |

### Зона приёма

| Команда | Действие |
|---------|----------|
| `reception set <id>` | Установить зону приёма в позицию игрока. |
| `reception radius <id> <blocks>` | Радиус зоны (`blocks` 1…128). |

### Каталог предметов

| Команда | Действие |
|---------|----------|
| `additem <pool> <cost> [quantity] [category] [модификаторы…]` | Захватить предмет из руки (полный NBT/TACZ) и добавить в `items.json`. |
| `removeitem <id>` | Удалить предмет из `items.json` (legacy-файлы `items/` удаляются вручную). |

**Модификаторы** `additem` — это хвост `ключ=значение` через пробел (значения с пробелами — в кавычках).
Ключи переопределяют позиционные аргументы:

| Ключ (синонимы) | Что задаёт |
|-----------------|------------|
| `name` (`displayname`) | `displayName` |
| `id` | id записи |
| `quantity` (`qty`) | штук за покупку |
| `category` (`cat`) | `displayCategory` |
| `maxwithdraw` (`max`, `maxperwithdraw`) | `maxPerWithdraw` |
| `refund` (`refundvalue`) | `refundValue` |
| `roles` (`role`) | `allowedRoles` (через запятую) |
| `teams` (`team`) | `allowedTeams` (через запятую) |
| `rank` (`minrank`) | `minRank` |
| `rolelock` (`rolelocked`) | `roleLocked` (`true`/`1`/`yes`/`да`) |
| `lockmode` (`lock`) | `lockMode` (`auto`/`use`/`hold`) |

Пример:

```text
/pjm warehouse additem weapon 3 1 weapon name="АК-74М" roles=assault,sso rank=private rolelock=true
```

### Личная квота (бюджет)

Управление личным бюджетом конкретного игрока (потолок берётся из его ранга):

| Команда | Действие |
|---------|----------|
| `budget <player> get` | Показать текущую квоту `остаток/потолок` (или «безлимит»). |
| `budget <player> full` | Восполнить квоту до потолка ранга. |
| `budget <player> set <n>` | Выставить остаток (зажим в `0…потолок`). |
| `budget <player> add <n>` | Добавить к остатку (не выше потолка). |

Для OP-игрока и безлимитного ранга команда сообщает, что квота безлимитна, и ничего не меняет.

### NPC-кладовщик

Действуют на **ближайшего NPC** в радиусе 12 блоков (кроме `spawn`):

| Команда | Действие |
|---------|----------|
| `npc spawn <warehouseId>` | Заспавнить NPC на позиции игрока, привязать к складу. |
| `npc skin <skinId>` | Сменить скин. |
| `npc team <team\|none\|all>` | Ограничить склад командой (`none`/`all` — снять). |
| `npc warehouse <id>` | Перепривязать NPC к другому складу. |
| `npc limit <amount>` | Лимит выдачи за раз (`0…256`, `0` — из предмета). |
| `npc cooldown <ticks>` | Задержка выдачи (`0…72000` тиков). |
| `npc categories <list\|all>` | Какие категории выдаёт (через запятую/пробел; `all` — все). |

## Перезагрузка конфигов

После правок `items.json`, `crates/` или секции `warehouse` перезагрузить без рестарта:

```text
/pjm config reload warehouse   # каталог предметов + ящики + флаги склада
/pjm config reload all         # все подсистемы
```

> Каталог и ящики также перезагружаются автоматически на `ServerStartedEvent`.
